## 2. Viết lại CombatResolver — Stack-based Combat

### Vấn đề cũ
Combat chạy tuần tự từng slot. Unit chết bởi ability (DealDamage, Destroy) không trigger OnDeath/OnAllyDeath/Reborn vì `HandlePotentialDeath` chỉ được gọi sau `ExecuteClash`.

### Kiến trúc mới

**`AttackIntent` struct** — tách bước "thu thập ý định" ra khỏi bước "thực thi":
```
BuildAttackIntents() → List<AttackIntent> (FIFO, slot 0→5, player trước enemy)
```

**`DeathEvent` struct + `Stack<DeathEvent>`** — mọi cái chết đều đi qua đây:
```
ScanAllBoardsForNewDeaths() → RegisterDeath() → deathStack.Push()
FlushDeathStack() → Pop → OnDeath → OnAllyDeath → CleanupBoard (Reborn/null)
```

**Flow `ResolveTurn`:**
```
1. ApplyTribeSynergies (cả 2 bên)
2. StartOfBattle + Aura → ScanDeaths → FlushDeathStack
3. for round 0..49:
   A. BuildAttackIntents
   B. foreach intent: ExecuteClash → ScanDeaths (ngay sau mỗi đòn)
   C. FlushDeathStack
   D. if IsSideEliminated → break
4. RecordSnapshots
```

**`FlushDeathStack` loop:**
```
while hasMore:
  while deathStack.Count > 0:
    OnDeath(victim) → ScanDeaths
    foreach ally (snapshot): OnAllyDeath → ScanDeaths
  CleanupBoard(Reborn/null) × 2 bên
  ScanDeaths → hasMore = deathStack.Count > 0
```

## 3. Bug Critical đã fix

### BUG 1 — Unit chết bởi ability không trigger OnDeath
**Root cause:** `HandlePotentialDeath` chỉ được gọi từ `ExecuteClash`.  
**Fix:** `ScanAllBoardsForNewDeaths` + `FlushDeathStack` — phát hiện mọi cái chết từ mọi nguồn.

### BUG 2 — `AddAbility` magic mutate shared `CardDefinition`
**Root cause:** `CardDatabase.GetCard()` trả về reference trực tiếp. `unit.Data.abilities.Add(ab)` sửa global data.  
**Fix:** `CardDefinition.Clone()` tạo bản sao với `new List<AbilityData>(abilities)`. `CardInstance` constructor dùng `data.Clone()`.

### BUG 3 — Tribe synergy +1 HP bị overwrite bởi `ApplyGrowth`
**Root cause:** `ApplyGrowth` recalculate `currentHP = baseHP + permanent + growth` → xóa +1 HP từ tribe.  
**Fix:** Đổi sang incremental: `target.currentHP += hpGain`.

### BUG 4 — `SummonUnit` overwrite unit đang chờ Reborn
**Root cause:** `board[i] == null || board[i].IsDead` không phân biệt unit chờ Reborn.  
**Fix:** Two-pass search — ưu tiên `null`, sau đó `IsDead && onDeathProcessed && !isReborn`.

## 6. Thiết kế mới: `conditionCount`

**Ý nghĩa:** Ability chỉ thực thi hiệu ứng vào đúng lần thứ N, 2N, 3N...

| `conditionCount` | Hành vi |
|---|---|
| `0` | Kích mỗi lần (mặc định) |
| `3` | Kích lần 3, 6, 9... |

**Implementation** (`AbilityEngine.TriggerAbility`):
```csharp
// Luôn đếm (trước đây chỉ đếm khi triggerLimit > 0)
source.abilityTriggerCounts[i]++;
// Skip effect nếu chưa đủ N lần
if (ability.conditionCount > 0 && source.abilityTriggerCounts[i] % ability.conditionCount != 0) continue;
```

**Kết hợp `triggerLimit` + `conditionCount`:**
- `conditionCount=3, triggerLimit=6` → kích lần 3 và lần 6 (2 lần tổng)
- `conditionCount=3, triggerLimit=3` → kích đúng 1 lần (lần 3)

> **Lưu ý:** `triggerLimit` đếm tất cả lần trigger (kể cả lần bị conditionCount skip). Số lần effect thực thi thực tế = `floor(triggerLimit / conditionCount)`.

## 7. Thiết kế mới: `TriggerAbility` effect

**Trước:** Kích toàn bộ abilities của target với mọi trigger gốc.  
**Sau:** Chỉ replay `OnDeath` abilities của target, lặp `scaleFactor` lần.

```csharp
for (int r = 0; r < scaleFactor; r++)
    TriggerAbility(TriggerType.OnDeath, target, directEnemy, allyBoard, enemyBoard);
```

**Cách dùng:** `StartOfBattle → TriggerAbility → target=SomeAlly` — "kích nổ" deathrattle của đồng minh trước trận. Lặp 1/2/3 lần theo merge level.

> **Ràng buộc:** Mọi ability có `effect = TriggerAbility` nằm trên `OnDeath` trigger **phải** có `triggerLimit >= 1` để tránh đệ quy vô hạn.

## 8. Fix cơ chế Reborn re-grant

**Cơ chế đúng:**
1. Unit có Reborn bị đòn chí mạng → sống với 1 HP, `hasRebornUsed = true`
2. Bị đòn chí mạng tiếp → chết thật
3. **Nếu** đồng minh grant `GiveBuff: isReborn = true` → Reborn được "nạp lại", có thể dùng lại

**Fix** (`AbilityEngine.ExecuteEffect.GiveBuff`):
```csharp
if (ability.isReborn) {
    target.isReborn = true;
    target.hasRebornUsed = false; // Reset cho phép dùng lại
}
```

## 9. Hệ thống Pre-combat Snapshot

**Vấn đề:** Unit được summon trong combat tồn tại qua turn sau (vi phạm luật cứng: "turn sau phải là đúng unit của end-Shop-Phase").

**Giải pháp:**
```
StartCombatPhase():
  → SnapshotPreCombatBoard()  ← chụp (slotIdx, CardInstance)[] TRƯỚC combat

EndCombatAndPrepareNextTurn():
  → RestorePreCombatPlayerBoard()
      Xóa tất cả playerSlots children
      Recreate chỉ từ snapshot (ResetStats → Instantiate)
  → SyncBoards()  ← sync playerBoard từ UI vừa rebuild
```

**Đảm bảo:**
- Summoned units trong combat KHÔNG có trong snapshot → bị xóa khi restore
- Stat không bao giờ giảm: `growthATKBonus`/`permanentATKBonus` chỉ tăng, `ResetStats()` formula dùng cả hai

## 11. Ràng buộc Thiết kế (Design Constraints)

| # | Ràng buộc | Lý do |
|---|-----------|-------|
| C1 | `triggerLimit` = tổng trigger (kể cả skip), không phải tổng effect | Tránh double-count; số effect = `floor(limit / condition)` |
| C2 | `OnDeath: TriggerAbility` phải có `triggerLimit >= 1` | Không có limit → đệ quy vô hạn khi `TriggerAbility` effect replay OnDeath |
| C3 | `Reborn` passive chỉ dùng 1 lần trừ khi được grant lại | `hasRebornUsed` chỉ reset khi nhận `GiveBuff: isReborn` từ unit khác |
| C4 | `conditionCount` chỉ có tác dụng với trigger lặp nhiều lần trong combat | `StartOfBattle`/`OnDeploy` chỉ kích 1 lần → `conditionCount > 1` không bao giờ fire |
