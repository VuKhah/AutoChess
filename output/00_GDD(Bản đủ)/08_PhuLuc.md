## Phụ lục B — Bảng Định Nghĩa 37 Gene

Chromosome 37-gene được tổ chức thành **9 nhóm chức năng**, mỗi nhóm kiểm soát một lớp quyết định riêng biệt trong vòng lặp `DecidePrepPhase`. Cách phân nhóm không phải tuỳ tiện — nó phản ánh đúng thứ tự ưu tiên mà một người chơi có kinh nghiệm nghĩ đến khi ra quyết định: trước hết đánh giá chỉ số (nhóm 1), rồi keyword (nhóm 2), rồi ability (nhóm 3–4), rồi ngữ cảnh đội hình (nhóm 5–6), cuối cùng mới đến chiến lược dài hạn (nhóm 7–9). ## Phụ lục D — Code Snippets Quan Trọng

### D.1 Hàm Đánh Giá Lá Bài — BotAgent.Evaluate()

Hàm cốt lõi kết nối chromosome với game state, được gọi trong mọi quyết định mua. Chú ý dòng cuối: chia cho `cost` rồi nhân `(1 + genes[3])` — đây là nơi chromosome quyết định bot có ưu tiên "rẻ-mà-tốt" hay sẵn sàng trả cao cho unit mạnh:

```csharp
private float Evaluate(CardDefinition c)
{
    float score = c.baseATK * brain.genes[0]
                + c.baseHP  * brain.genes[1]
                + (c.tier - 1) * brain.genes[2] * 5f

                + (c.hasTaunt     ? brain.genes[4] * 10f : 0)
                + (c.hasReborn    ? brain.genes[5] * 12f : 0)
                + (c.hasSafeguard ? brain.genes[6] * 8f  : 0);

    if (c.abilities != null)
    {
        int sameTribeCount = board.Count(u => u != null
                             && u.Data.tribe == c.tribe && !u.IsDead);
        foreach (var ability in c.abilities)
        {
            float tw = TriggerWeight(ability.trigger);
            float ew = EffectWeight(ability.effect);
            float ctx = (ability.trigger == TriggerType.OnAllyDeath
                      || ability.trigger == TriggerType.OnAllySummon
                      || ability.trigger == TriggerType.OnAllyReborn)
                      ? Mathf.Clamp01(sameTribeCount / 2f) : 1f;

            score += tw * ew * 10f * ctx;
            if (ability.isEscalating) score += tw * ew * 3f;
        }
    }

    score += sameTribeCount * SynergyWeight(c.tribe) * 4f;
    score += copiesOnBoard  * brain.genes[21] * (copies == 2 ? 16f : 8f);
    score += (c.hasTaunt ? emptyFront * brain.genes[22] * 2f : 0);

    return score / Mathf.Max(1, c.cost) * (1 + brain.genes[3]);
}
```

Một điểm tinh tế: `ctx = Clamp01(sameTribeCount / 2f)` áp dụng cho OnAllyDeath/Summon/Reborn — triggers phụ thuộc vào "có đồng minh để react không". Bot tự nhiên học rằng các triggers này gần như vô dụng khi chỉ có một unit trên sân, và cực kỳ giá trị khi đã có 3–4 đồng minh cùng tribe. Đây là context-awareness được encode trực tiếp vào hàm đánh giá mà không cần thêm gene.

### D.2 Hàm Fitness Một Trận — GameSimulator.ScoreFromA()

Công thức fitness hiện tại ưu tiên tuyệt đối kết quả thắng/thua — win premium 290 điểm (300−10) so với 85 điểm của phiên bản trước (120−35). LateScore và cardScore là tín hiệu phụ, chỉ có ý nghĩa khi kết quả và HP gần nhau:

```csharp
private static float ScoreFromA(int result, int hpA, int hpB, int turns,
                                  float lateScoreA, float lateScoreB,
                                  float cardScoreA, float cardScoreB)
{
    float score = result > 0 ? 300f    // thắng
                : result == 0 ? 100f   // hòa
                : 10f;                 // thua

    score += hpA * 8f;
    score -= hpB * 4f;
    score += Mathf.Max(0f, lateScoreA)                          * 0.06f;
    score += Mathf.Clamp(lateScoreA - lateScoreB, -300f, 300f)  * 0.04f;
    score += Mathf.Max(0f, cardScoreA)                          * 0.035f;
    score += Mathf.Clamp(cardScoreA - cardScoreB, -250f, 250f)  * 0.025f;
    if (turns >= 12)
        score += (turns - 11) * 2f;

    return Mathf.Max(1f, score);
}
```

Bonus `(turns - 11) * 2f` (áp dụng từ lượt 12 trở lên) khuyến khích bot xây đội hình bền vào late game thay vì all-in sớm — đây là đảo ngược so với phiên bản cũ (`(MaxTurns - turns) * 2f`), nơi thắng nhanh được thưởng nhiều hơn.

### D.3 Tournament Selection Thích Nghi — GATrainer

Kích thước tournament tăng theo tiến trình (`progress` từ 0.0 → 1.0) để áp lực chọn lọc tăng dần, tránh hội tụ quá sớm ở đầu training:

```csharp
private static int CurrentTournamentSize(float progress)
{
    if (progress < 0.35f) return 3;   // early: diversity cao
    if (progress < 0.75f) return 4;   // mid
    return 5;                          // late: hội tụ nhanh
}

private Chromosome TournamentSelect(List<Chromosome> pop, float progress)
{
    int k = CurrentTournamentSize(progress);
    var contestants = Enumerable.Range(0, k)
        .Select(_ => pop[Random.Range(0, pop.Count)])
        .ToList();
    return contestants.MaxBy(c => c.fitness);
}
```

Kỹ thuật này gọi là *annealing selection pressure*: đầu training ưu tiên khám phá (k=3 ít loại thẳng), cuối training ưu tiên khai thác (k=5 chỉ giữ chromosome tốt nhất trong mỗi tournament). Tương tự simulated annealing nhưng áp dụng lên áp lực chọn lọc thay vì nhiệt độ.

### D.4 Elitism Bốn Tầng — GATrainer

Bảo toàn top global và top mỗi archetype riêng — đảm bảo ngay cả khi một archetype có fitness thấp hơn, top-2 đại diện của nó vẫn được giữ lại:

```csharp
// Tầng 1: Elite tổng — số lượng tăng từ ~popSize/18 đến ~popSize/8 theo tiến trình
AddTopClones(nextGen, population, c => true, eliteCount);

// Tầng 2–5: Elite theo archetype — luôn bảo toàn 2 đại diện tốt nhất mỗi loại
AddTopClones(nextGen, population, IsBabylon,   2);
AddTopClones(nextGen, population, IsNiles,     2);
AddTopClones(nextGen, population.OrderByDescending(SummonerScore),   2);
AddTopClones(nextGen, population.OrderByDescending(ResilientScore),  2);
```

Không có elitism per-archetype, Babylon có thể bị xóa hoàn toàn trong giai đoạn Niles Domination (thế hệ 33–65 ở lần training cũ, và Other Domination 100–150 ở lần training mới) — và kết quả cuối sẽ không có babylonBot đủ tiêu chuẩn để chọn. Tầng 2–5 là "bảo hiểm diversity" không thể bị override bởi selection pressure.

### D.5 Death Stack — CombatResolver.FlushDeathStack()

Xử lý cái chết theo thứ tự LIFO (Last In First Out) bằng stack, đảm bảo chain death không bị interleave — unit B chết do OnDeath của A được xử lý hoàn toàn trước khi A tiếp tục:

```csharp
private void FlushDeathStack(List<CardInstance> pBoard, List<CardInstance> eBoard)
{
    while (deathStack.Count > 0)
    {
        var evt = deathStack.Pop();
        var (victim, killer, vBoard, kBoard) = evt;

        if (victim.isReborn && !victim.hasRebornUsed)
        {
            victim.ReviveDefault();
            engine.BroadcastAllyEvent(TriggerType.OnAllyReborn, victim, vBoard, kBoard);
        }
        else
        {
            int slot = vBoard.IndexOf(victim);
            if (slot >= 0) vBoard[slot] = null;

            engine.TriggerAbility(TriggerType.OnDeath, victim, killer, vBoard, kBoard);
            engine.BroadcastAllyEvent(TriggerType.OnAllyDeath, victim, vBoard, kBoard);
        }

        while (engine.HasPendingSummons)
            engine.ProcessNextPendingSummon();
    }
}
```

Vòng `while (engine.HasPendingSummons)` bên trong vòng lặp chính là điểm quan trọng: unit được summon từ OnDeath (ví dụ Tawaret → 2 Medjed) phải được xử lý *trước* khi pop cái chết tiếp theo ra khỏi stack. Nếu không có vòng lặp này, Medjed có thể xuất hiện sau khi Anubis đã xử lý OnAllyDeath — và miss trigger.
